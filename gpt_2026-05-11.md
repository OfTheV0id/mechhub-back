## 当前关键接口语义

### `POST /api/auth/register`

- 请求字段: `email`, `password`, `name`, `role`。
- `role=student`:发学生邮箱验证邮件。
- `role=teacher`:发管理员审批邮件。
- 不登录、不 set cookie。

### `GET /api/auth/verify?token=xxx`

- student 邮箱验证。
- token kind: `verify`。
- 成功后 `verified=true`。
- 不登录、不 set cookie。

### `GET /api/auth/approve-teacher?token=xxx`

- teacher 管理员审批。
- token kind: `teacher_approval`。
- 成功后 `verified=true`。
- 不登录、不 set cookie。
- 安全提醒:不要把这个接口改成 set teacher cookie。审批链接在 admin 邮箱里,如果 set cookie 会让 admin 浏览器变成 teacher 登录态。

### `POST /api/auth/login`

- 要求账号 `verified=true`。
- 成功后 set `session_id` cookie。
- 返回 `userdata`。

### `POST /api/user/update-profile`

- 需要登录 cookie。
- 只改 `name`。
- 成功后返回刷新后的 `userdata`。

### `POST /api/user/change-password`

- 需要登录 cookie。
- 成功后更新密码,删除该用户所有 session。
- handler 会清当前 cookie。
- 前端应清登录态并回登录页。

## 踩坑和经验

1. 用户会在过程中纠正产品语义,要以最新一句为准。

一开始他说 student 不需要邮箱验证,后来明确指出 student register 只是发出验证邮件。最终应以当前实现为准:student 需要邮箱验证,teacher 需要 admin 审批。

2. register 和 verify/approve 不能混为登录。

最终语义是:注册、邮箱验证、teacher 审批都不登录。只有 `/api/auth/login` 会 set cookie。

3. teacher approve 不应该 set cookie。

审批链接发给 admin,如果审批时 set teacher session,admin 浏览器会变成 teacher 登录态。这是明显安全/体验问题。

4. 返回字段要统一。

登录、`/api/user/me`、`/api/user/update-profile` 都应该返回同一套公开用户字段。当前为:

- `id`
- `email`
- `name`
- `role`
- `avatar_url`
- `verified`
- `created_at`

5. Postman 路径可能被用户手动调整。

不要硬编码旧文件夹名。改 Postman 前先 glob 当前实际路径。

6. 不要覆盖未验证账号密码。

这条沿用 `HANDOFF.md` 的安全结论。重复注册未验证账号时只刷新 token/重发邮件,不要改 password hash。

7. 改密后踢掉所有 session 是有意设计。

不要为了“用户体验”去掉 `sessions.DeleteByUser`。

8. 配置新增必须同步 `.env.example`。

本轮新增 `ADMIN_EMAILS`,已经同步。
